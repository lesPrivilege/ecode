# Note — 视图化 context：意图声明 + 按读取次的语义视图（设计种子）

2026-07-05。状态：**种子,不实现**。等 DF0 数据（分诊 ≥5 条）复盘后裁定是否立项。
来源：dogfooding 首日观察（replacement 后反复重读）引出的两步推演。

## 核心翻转（两个,递进）

1. **假设 → 契约**：现行方案赌 first-reply 假设（模型读完会把语义写进叙述）。
   改为模型在 tool call 时**显式声明**留存意图——`retention: semantic |
   verbatim | routing` 作为 read/bash 参数（call-time,进 append-only log,
   前缀友好,近零成本）;post-hoc 标记块作补充。harness 读声明、路由不同
   压缩策略。**声明是 hint,确定性守卫保留最终裁量**——模型可能误判未来需求。

2. **重读 = 病理 → 重读 = 取视图**：文件是磁盘上的免费稳定底层,context 只持有
   当前 loop 相位的视图。每次读带意图产出该意图的语义,服务当前 loop,用后
   激进压缩;下一相位要不同语义就再读取新视图。真相源在文件系统,`path#hash`
   寻址。分诊 (a) 类中「带新意图的定向重读」应重分类为正常操作;病理只剩
   无进展的全文反复读。

## 语义台账（工程形态）

`path#hash → [(turn, 意图, 视图摘要)]`。压缩时旧视图降格、台账保留;
文件被 edit（hash 变）→ 该 path 全部视图失效（hashline 机制现成）。
台账轻量注入 context,模型可见「此文件已有哪些视图」,重读决策有据可依。
台账自身需要 retention 策略（防自我膨胀）。

## 成本模型改写（可测假设）

旧账:重读 = 纯支出。新账:净流 = 持有全文 × N turn 的累计成本 −
按需定向重读的一次性成本。假设:5-28 负案（47% 省 /124 重读）的病根是
无定向全文重读;若重读是带 offset/limit 的定向取视图,同一负区场景可能翻正。
G2-D1/D2 负区探针加视图化对照臂即可检验（后续轮,非 round 1）。

## 校准指标（免费副产品）

「声明 disposable 但后来重读」率 = 模型对自身信息需求的自知程度。
分诊表加一列（声明意图 vs 实际重读）即可测。跨模型跑同一套,校准率差异
= 能力等高线实测点（论文素材;「工作语义建立依赖权重」的可量化面）。

## 依赖与边界

- pi extension 能否扩展内建 read 的参数 schema——待验证（可能需包装工具）。
- 声明质量绑定 base model 能力（墙二）:校准率低则整个方案退化为
  现行确定性默认,优雅降级,不会更坏。
- 视图/台账自身的 token 开销要进净流账,防「为省而费」。

## 触发条件

DF0 分诊数据显示 (b) 类（摘要结构缺陷）占优 → 先做结构化摘要（更便宜）;
(a) 类占优且多为「新意图定向重读」→ 本方案升格立项,从台账+意图参数最小版开始。
